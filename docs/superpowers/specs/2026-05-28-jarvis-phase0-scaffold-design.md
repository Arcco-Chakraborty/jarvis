# JARVIS — Phase 0 (Scaffold) Design

**Date:** 2026-05-28
**Status:** Approved — ready for implementation planning
**Scope:** Phase 0 only (PROJECT.md §8). Phases 1–5 are out of scope here.
**Source of truth:** `PROJECT.md`. This spec refines *how* Phase 0 is built; if it ever
conflicts with PROJECT.md, PROJECT.md wins.

---

## 1. Goal

Ship a runnable orchestrator skeleton whose only behavior is `GET /health → {ok:true}`,
with the device registry seeded from the §4 channel map and the existing ESP32 adapter
wired in and polling the live board. Each later phase builds on this foundation.

**Acceptance:** `npm install` succeeds → `npm test` is green → `npm start` boots →
`curl localhost:3000/health` returns `{"ok":true}`. Bonus (board is live):
`curl localhost:3000/state` shows real cached relay states. Repo pushed to a private
GitHub repo `jarvis` with no secrets committed.

## 2. Locked decisions

These were settled during brainstorming and are not open for re-litigation in the plan:

| Decision | Choice | Reason |
|----------|--------|--------|
| Scope | Spec-faithful scaffold (PROJECT.md §8 Phase 0) | Exactly what the phase calls for; makes Phase 1 a small step |
| ESP32 base URL | `http://192.168.0.202` | User-provided; verified reachable via `GET /state` on 2026-05-28 |
| Module system | ESM (`"type":"module"`) | The adapter is ESM; one module system across the orchestrator |
| Env loading | Node 22 native `--env-file=.env` | No `dotenv` dependency — keeps deps lean (PROJECT.md §9) |
| Device IP location | Only in `.env` (gitignored), never in committed code | Never commit device IPs/secrets |
| ESP32 polling in P0 | Yes — `startPolling()` + `GET /state` debug | Matches §5.1 boot sequence; verifies adapter↔firmware now |
| Empty dir stubs | Create `voice-service/`, `pc-agent/`, `deploy/` with `.gitkeep` | Visible §10 skeleton; matches checkpoint's "create layout" step |
| GitHub | Create private `jarvis` repo now and push | User choice |

**Networking note:** the host is `192.168.1.167` (subnet `192.168.1.x`); the ESP32 is
`192.168.0.202` (a *different* /24). Despite the mismatch, the board is reachable from the
host (cross-subnet routing works on this LAN). Do not assume board and host share a subnet.

## 4. Component: `orchestrator/config.js`

Reads `process.env` (populated by `--env-file=.env`) and exports a frozen `config`:

```js
export const config = {
  port: Number(process.env.PORT ?? 3000),
  dbPath: process.env.DB_PATH ?? 'orchestrator/db/jarvis.db',
  esp32: { baseUrl: process.env.ESP32_BASE_URL },   // REQUIRED — throw if missing
  pcAgentToken: process.env.PC_AGENT_TOKEN ?? '',    // unused until Phase 3
  geminiApiKey: process.env.GEMINI_API_KEY ?? '',    // unused until Phase 4
};
```

- `ESP32_BASE_URL` is **required**: if absent, throw a clear error
  (`"ESP32_BASE_URL is required — set it in .env"`). No real IP is baked into this
  committed file.
- `pcAgentToken` / `geminiApiKey` default to `''`; they are not validated in Phase 0
  because no code path uses them yet.

## 7. Component: `orchestrator/db/registry.js`

```js
export function openRegistry(dbPath = config.dbPath) { /* ... */ }
```

On open: instantiate `better-sqlite3` at `dbPath` (creates the file if absent), run
`schema.sql` via `db.exec`, then seed idempotently. Returns an API object:

- `getBoard()` → the `smartswitch` device row `{ id, name, type, base_url }`.
- `getSwitchNamesByChannel()` → array of 8 switch names ordered by `channel` 0–7,
  for passing into `Esp32Switch({ names })`.
- `close()` → close the DB handle (used by tests).

**Seeding (idempotent, `INSERT OR IGNORE`):**

1. Device: `('smartswitch', 'esp32_switch', config.esp32.baseUrl)`.
2. The 8 switches, mapped to that device id, names matching the adapter's lowercase
   `DEFAULT_NAMES`:

| channel | name          | group_name |
|---------|---------------|------------|
| 0       | `fan 1`       | fans       |
| 1       | `fan 2`       | fans       |
| 2       | `tubelight`   | lights     |
| 3       | `spotlight`   | lights     |
| 4       | `rgb light`   | lights     |
| 5       | `night light` | lights     |
| 6       | `socket`      | other      |
| 7       | `spare`       | other      |

The registry is the **single source of truth for names**; `server.js` feeds those names
into the adapter so the two never drift (resolves the PROJECT.md "naming gotcha").

> Re-seeding an existing DB must be a no-op: still one `smartswitch` device, still
> exactly 8 switches. Note the seeded `base_url` is captured at first seed; changing
> `ESP32_BASE_URL` later updates `.env`/the adapter but not the stored row — acceptable
> for Phase 0 (the row is unused for routing until Phase 1).

## 8. Component: `orchestrator/server.js`

Split for testability:

- **`buildApp({ esp32 })`** → returns the configured Express app, no network binding.
  - `GET /health` → `{ ok: true }`
  - `GET /state` → `{ ok: true, smartswitch: esp32.snapshot(), online: esp32.online }`
- **`main()`** — wires real dependencies and listens:
  1. `const registry = openRegistry(config.dbPath)` (creates + seeds the DB)
  2. `const esp32 = new Esp32Switch({ baseUrl: registry.getBoard().base_url, names: registry.getSwitchNamesByChannel() })`
  3. `esp32.startPolling()`; attach listeners: `offline` → `console.warn`,
     `online` → `console.log`, `change` → `console.log`
  4. `buildApp({ esp32 }).listen(config.port, …)`

This matches the §5.1 boot sequence (load registry → instantiate board → start polling →
start HTTP server).

## 9. Testing (TDD)

Built-in `node:test` + global `fetch`; no extra test dependencies.

- **`registry.test.js`** — open a temp/`:memory:` DB:
  - the four tables exist after open;
  - seeding produces exactly 8 switches with the correct `(name, channel, group_name)`
    and exactly one `smartswitch` device;
  - opening/seeding the same DB twice is idempotent (still 8 switches, still 1 device).
- **`server.test.js`** — `buildApp({ esp32: stub })` where the stub returns a fixed
  `snapshot()` and `online`:
  - `GET /health` → 200 `{ ok: true }`;
  - `GET /state` → 200 containing the stub snapshot and `online`.
  - Boot via `app.listen(0)`, read the assigned port, `fetch`, close.

The live-board `curl` is a manual bonus check, not an automated test.

## 10. Git / GitHub

1. `git init` (done during brainstorming for the spec commit).
2. Write `.gitignore` **before** staging anything that could touch secrets.
3. Commit the scaffold. **Never stage `.env` or `*.db`.**
4. User runs `gh auth login` (interactive) — flagged at the right moment.
5. `gh repo create jarvis --private --source=. --remote=origin --push`.

## 11. Risks

- **`better-sqlite3` native build.** On Node 22 it should fetch a prebuilt binary. If the
  prebuilt is unavailable, npm falls back to compiling via node-gyp, which needs
  `build-essential` and a compatible Python (the host has Python 3.14, which is very new
  and may not satisfy node-gyp). *Mitigation at execution:* if `npm install` fails, either
  `sudo apt install -y build-essential` (user, needs sudo) or pin a `better-sqlite3`
  version that ships a Node-22 prebuilt.
- **`--env-file` requires `.env` to exist.** Phase 0 creates it, so `npm start` works; a
  fresh clone must `cp .env.example .env` first (documented in README).

## 12. Out of scope (later phases)

Rule matcher / `POST /command` (Phase 1), voice service (Phase 2), PC agent + music
(Phase 3), Gemini fallback (Phase 4), dashboards/expansion (Phase 5). No message broker,
no firmware changes, no `/toggle` — ever (PROJECT.md §2).
